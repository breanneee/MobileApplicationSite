History

A brief overview of how mobile apps came about.

  • 1984

    First Mobile Phone

    Motorola releases the first commercially available mobile phone, the Motorola DynaTAC 8000X. It weighed nearly two pounds and cost close to $4,000 and had a battery that would last for a thirty minute conversation.

  • 1998

    Wireless Markup Language

    In 1998 the WML Standard was released. WML was a language that used Wireless Application Protocol, or WAP, to deliver content in a more mobile-friendly way to mobile devices.

  • Late 1990's

    Use of Gameboy and PDA's

    As users began to see the mobile friendly gaming devices such as the Gameboy develop, and the corporate friendly devices such as the PDA, it became obvious that an all-in-one solution was the future.

  • 2007

    Apple announces iPhone and App Store

    On January 9, 2007 Apple announced the release of the first iPhone which revolutionized mobile apps with it's enlarged display and touch screen. Furthermore, the App store revolutionized the mobile application industry as it became the first widely known platform for third-party developers to host their apps. At first release, the App Store was only open to web applications, but it was not long before developers had the ability to create native applications.

  • 2008

    Google launches Android Market

    On August 28, 2008 Google announced the Android Market which is their version of Apple's App Store, and it was made available to the public on October 22.

  • 2014

    Over 1.2 Million Apps

    Today, app developers are everywhere, from big corporations to individuals. There are over 1.2 million applications in Apple's App Store alone, not to mention the rest.

iOS

by Apple

Introduction

iOS was released by Apple in June 2007 for its recently-released smartphone, the iPhone. Since then, the range of devices has grown to include the iPad and Apple Watch. The current version of iOS is iOS 8. The basic framework behind iOS is shown in the image below. It follows an MVC paradigm. The bulk of typical coding takes place in Views, ViewControllers, and the Application Delegate.

A View is an individual UI element, like a button or text field. These can be created in code or graphically through the Interface Builder. The visual representation is connected with the code through an @IBOutlet tag, which can be placed directly into the code or generated by CTRL+clicking and dragging from the UI element to a location in code (Assistant view).

A subclass of ViewController is used to manage the various Views present in a screen that the user is seeing. ViewController objects are also created from Interface Builder.

The Application Delegate, or app delegate, is used to manage the singleton UIApplication object when certain state changes occur (such as the app going into the background, or a notification coming in from outside the app). The app delegate is created by Xcode when a new project is made.

In the picture above are some of the state changes that the app delegate manages. Each of these state changes has a corresponding method that can be used to manage the transition. The methods are named clearly and appropriately. For example, if the app moves from the 'inactive' state to the 'active' state, the method 'applicationDidBecomeActive' will be called. The developer can simply place any code that needs executed during this transition into that method.

Development Environment

Interface Builder has been mentioned a couple times so far in the description of various architectural components. It is just one of many features in Xcode, Apple's IDE for iOS development.

Interface Builder provides a way to graphically lay out the views in each ViewController and to see plainly the different transitions and interactions between ViewControllers. References to UI elements can be managed, and actions can easily be added to UI interactions like button clicks and gestures.

In "Assistant" mode, @IBOutlet connections between UI elements and the code can be created with just a CTRL+click and a drag of the mouse.

Another feature that can help the developer graphically is the View Debugger, which presents UI elements in 3D, making it easier to see the different layers of views presented to the user. While none of these graphical features are necessary for development, they can speed things along and make the entire process more enjoyable if used effectively.

Swift

Some of the newest features added to the iOS development workflow in iOS 8 concern the Swift programming language. Swift has some exciting new features compared with the previous status quo of Objective-C, including inferred types, closures, the 'let' keyword, and tuples.

Inferred types provide a Javascript-style system where assigning the type of a variable is no longer mandatory. 

var myString="helloWorld"

Closures can be used to pass anonymous functions as parameters or in the queue used by iOS' asynchronous task management system. 

reversed = sorted(names, { (s1: String, s2: String) -> Bool in
   return s1 > s2
})

The 'let' keyword can be used to define a constant. 

let numbers = [16, 58, 510]

Tuples can be used to contain multiple variables without the need to make a class. This essentially also allows for returning multiple values. 

func someFxn(array: [Int]) -> (num: Int, otherNum: Int) {
    var myNum = array[0]
    var myOtherNum = array[1]
    return (myNum, myOtherNum)
}

Another recent addition is the 'Playgrounds' feature. Playgrounds are a way to see the result of your code in an alternative format that can help illuminate what's going on behind the scenes. For example, the result of assigning something inside a loop can be plotted over time, as in the picture below. Another important application of Playgrounds is learning. Apple has started providing sample code in the format of ready-to-go Playgrounds, making it even easier to play with different elements of the completed code and analyze how the different pieces are working together.

Publishing

The Apple App Store is perhaps the most well-known medium for publishing and obtaining mobile apps, being the first major platform introduced as such. The typical publishing process consists of the following steps:

  • Sign up for the Apple Developer program ($99 USD/yr)
  • Set up provisioning profiles for development and testing on devices
  • Make sure all necessary icons, graphics, etc. are included according to Apple's Human Interface guidelines
  • Submit packaged-up app to Apple for review
  • Receive a response
  • Receive payments, if your app has a price

Advantages & Disadvantages

Advantages
  • Quick feedback in UI development
  • Long-standing ecosystem, documentation, and community
  • Decreased need to support many different types of devices' hardware
  • Large market share
Disadvantages
  • The Apple ecosystem is a "walled garden"
  • Setting up development devices can be more difficult that in Android
  • Joining the Developer Program costs money
  • The approval process and criteria for publishing in the App Store can be quite stringent

Android

Powered by Google

Background

The Android Operating System was released commercially for the first time in 2008 shortly after iOS and the iPhone launched. Since then the open source operating system has gone through various updates and is currently on version 5.0. The most current version stays with the 'sweet treats' inspired naming convention and is called Lollipop. Because of the open source nature of the project, the Android OS runs in many variations on a wide multitude of devices. This may have contributed to the fact that 85% of all phones sold globally last year being Android based. The Android project is currently being developed jointly by Google and the Open Handset Alliance and the source code is written in a combination of C/C++ and Java. The framework they have developed can be visualized below where everything in blue was written in Java while everything else is C, C++ or a combination of the two.

Development Environments

Development environments can really impact the effectiveness and convenience of a developer. Fortunately for Android, there are several options to choose from. As far as IDEs go, the two main options are Eclipse with the Android Development Tools (ADT) plugin or Android Studio. Eclipse is a good option for those already accustomed to this IDE and has the advantage of a variety of plugins available. Android Studio on the other hand is still in its Beta stage however there have been good reviews already. It is based on IntelliJ IDEA and will be endorsed as the official IDE once development is complete. Both of these options have debuggers and profilers as well as built in emulators to test apps. A common complaint is that these emulators are quite slow, however, and many use a third party emulator called Genymotion which is built using VirtualBox.

Design Features

Following are some of the key design components of an Android application. When building any sort of app a developer will most likely run into all of these features and most likely use all of them too.

  • Activity – easy to think of this as a “screen” in an application

  • Layout – defined in an xml file; comes in different flavors (absolute, relative, linear)

  • Views – visible component of an activity; location defined in layout file; often have onClick handlers

  • Service – a background process that may not have a view associated

  • Widget – view only interface to service

  • Action Bar – introduced in 3.0; provides useful user options

  • Options Menu – resides in Action Bar and present by default; options defined in menu layout xml files

  • Fragments – reusable portions of UI; useful for supporting both small and large sized screens

  • Persistence – Data can be stored on internal phone storage or external SD card using SQLite

  • Content Providers – avenue for sharing data between other apps on device (like contacts or media)

The methods can be overridden by the developer to include calls to methods that need to be executed everytime that part of the lifecycle is entered (such as saving user data, or populating text fields, etc.)

Publishing

Publishing options have grown since the inital launch of Android. The Android Marketplace was the only platform to publish apps upon the initial release of the Android operating system back in 2008. Since then the name has changed to the Google Play Store and over 1.3 million apps are available for download there as well as music, videos and other media. Google moniters the apps listed on this platform for security and policy violations. Another nice feature is that unlike the Apple App Store, when a user downloads an app from Google Play Store, they are immediately notified of all the permissions the app requires. Recently Amazon Appstore has also shown to be a growing distribution option for developers of Android apps.

Advantages & Disadvantages

Advantages
  • Free to start developing
  • Large and growing user base
  • Less restrictive policies to distribute app
  • Multiple distribution platforms
Disadvantages
  • Registration fee to distribute app to Play Store
  • Large variety of devices and OS versions to support
  • Multiple distribution platforms
